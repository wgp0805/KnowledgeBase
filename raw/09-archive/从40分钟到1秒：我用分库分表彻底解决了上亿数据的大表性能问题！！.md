三年前，我的数据库还是个温顺的小绵羊。

订单表才三十万行，用户表不到十万，每次查询快得跟翻手机相册似的。我端着咖啡跟新来的实习生吹牛：“咱这架构，传家宝级别，再战五年不成问题。”

结果呢？一年后用户量翻了八倍，订单表直奔两千万。有次运营小姐姐让我帮忙拉一份“近三个月销量榜”，我点了查询，然后去接水、上厕所、看两眼朋友圈——回来一看，页面还在转圈。那杯接回来的水都凉透了，SQL 还没跑完。

双十一更刺激。用户下单后付款按钮一直灰着，客服群里炸了锅：“我付了三次钱，订单都哪儿去了？”大半夜我抱着笔记本蹲机房，看着日志里满屏的“Connection timeout”，感觉每个红色报错都在冲我做鬼脸。

第二天，技术总监拍着我肩膀：“要不咱试试分库分表？”

我当时内心是拒绝的。这玩意儿听起来就像“你心脏不好，要不咱换个猪心？”——又麻烦又吓人。

但后来硬着头皮搞完了，回头一看，其实没想象中那么玄乎。今天我就用说人话的方式，把分库分表那点事儿扒干净，顺便送你一套 Java 代码，让你明天就能动手。

## 一、你家的数据库，为啥跑不动了？

先别急着怪数据库。人家本来的设计是“精致小车”，你非要当重卡用，不趴窝才怪。

数据库跑不动，本质就两个原因： **数据太多** 和 **问的人太多** 。

### 1\. 数据太多 → 翻垃圾堆找针

你的表就像一个大衣柜。刚开始衣服少，你想找那件黑色羽绒服，一眼就瞄到了。等衣柜塞满几百件衣服，你就得一件一件翻，甚至把所有衣服倒出来才能找到。

数据库也一样。数据存在磁盘上，磁盘有个臭毛病——它喜欢 **连续读** （像看书一页页翻），讨厌 **跳着读** （像从第10页直接翻到第100页）。所以数据库把数据分成“数据页”，每页16KB，先读到内存再查。

当单表数据超过 **1000万行** ，问题就来了：

- 索引本身也变胖了。B+树从3层变成4层，原来3次磁盘IO能搞定的事，现在要4次，速度直接打七五折。
- 数据页散落在磁盘各个角落。查100条数据可能要读50个不同的页，磁盘来回寻道，慢得像老牛拉破车。

我一个朋友，用户表干到8000万行。运营要统计“2020年注册的广东用户数”，跑了40分钟没出结果，最后 DBA 把表锁了，导到 Excel 里用筛选功能统计的。你说这是人干的事吗？

### 2\. 问的人太多 → 十字路口大堵车

你想象一下，数据库就是一个十字路口，每个 SQL 就是一辆车。

平时车少，红绿灯指挥得挺好。一到晚高峰，所有车都挤在路上，谁也走不了。更气人的是，写操作（insert/update）像大卡车，路过的路口还要临时封路（锁表），后面的车只能干瞪眼。

数据库的连接池是有限的。MySQL 默认最大连接数 151，超过这个数的请求就得排队。我见过一个生鲜电商，大促时订单接口 QPS 冲到 5000，数据库只能扛 1200，剩下的 3800 个请求在排队，用户点了下单要等 10 秒才有反应，一半用户直接放弃支付。

这时候你可能会说：加 Redis 缓存啊，换更强服务器啊。

我试过。缓存只能存热点数据（比如最近1小时订单），查三个月前的订单照样崩。升级服务器更坑——从 8 核升到 16 核，QPS 从 1200 涨到 1500，花了十几万，性能才涨 25%，老板脸都绿了。

所以，当数据量和并发量双双突破阈值，分库分表就不是“要不要”，而是“再不搞就真没了”。

## 二、分库分表是啥？拆了衣柜和房间

别被术语吓到。分库分表的核心思想就四个字： **分而治之** 。

你衣柜乱，那就买几个收纳盒，按季节分：春装一个盒，夏装一个盒。这叫 **分表** （一个大表拆成多个小表）。

你房间不够大，那就再租一间房，把换季衣服搬到另一间。这叫 **分库** （多个数据库实例分担压力）。

### 2.1 分表（水平 vs 垂直）

**水平分表** ：按行拆。比如订单表，按时间拆成 `order_202401` 、 `order_202402`... 每个月的数据放一张表。查本月订单只查一张表，快如闪电。

**垂直分表** ：按列拆。比如用户表，把常用字段（id、昵称、手机号）放 `user_base` ，不常用字段（地址、爱好、身份证）放 `user_detail` 。这样 `user_base` 表变小了，查询常用信息更快。

我之前优化一个社交 APP，用户表 50 多个字段，登录只需要 id、手机号、密码。拆完后 `user_base` 只有 10 个字段，数据量少了 80%，登录查询从 200ms 降到 30ms，用户体验起飞。

### 2.2 分库

分表解决不了 CPU、内存、磁盘 IO 的瓶颈，因为所有表还在同一个数据库实例里。分库就是把小表分布到不同数据库服务器上，每台服务器只扛一部分压力。

比如按用户 ID 哈希分库：ID 尾号 0-4 去 `db1` ，5-9 去 `db2` 。用户每次操作只访问一个库，互不干扰。

通常我们会 **分库+分表** 一起做：先分库（比如2个库），每个库内再分表（比如12个月度表），一共 2×12=24 个小表。每个小表的数据量是原来的 1/24，每个库的并发压力是原来的 1/2，性能直接翻倍。

那么问题来了：我怎么知道数据在哪个库哪张表？难道要自己写 if-else 算？

不用。用 **中间件** ，比如 Sharding-JDBC。你写 SQL 还是 `select * from order where user_id=123` ，中间件自动帮你路由到正确的库和表，你完全不需要改业务代码。

## 三、分库分表三大大坑（我替你踩过了）

很多人以为分库分表是银弹，上了就万事大吉。我跟你说，坑多得能把你埋了。

### 坑一：分片键选错，数据全堆一个坑里

分片键就是拆分的依据。比如按 `user_id` 分， `user_id` 就是分片键。

我见过一个团队按“订单ID”哈希分表。结果订单ID是自增的，连续生成，哈希后大部分落到了几个分片上，其他分片几乎空的。大促时那几分片 CPU 爆满，其它分片在“睡觉”。这叫 **数据倾斜** 。

**怎么选分片键？两个原则：**

1. **高频查询字段** ：比如你经常查“某用户的订单”，那就按 `user_id` 分。这样查一个用户的数据只命中一个分片，不用跨片查。
2. **避免热点** ：比如按“商品ID”分，爆款商品的订单全挤到一个分片，会炸。可以改成“商品ID哈希 + 时间范围”双重分片，分散热点。

另外，分片键最好是 **不可变的** 。你要是按“手机号”分，用户换手机号，数据要搬家，折腾死人。

### 坑二：跨分片查询，查个数据跑断腿

你按 `user_id` 分了库，现在要查“2024年5月所有订单总额”，就必须去所有分片查 `order_202405` 表，再把结果加起来。就像你要统计全国肯德基的薯条销量，得一家家打电话问。

我见过一个报表系统，每天凌晨统计全量数据，要跨 16 个库、192 张表，原本单库跑 10 分钟，分库后跑了 2 小时，最后不得不把报表改成离线跑，用大数据平台算。

**解决方案：**

- **业务规避** ：少做全表统计。改成“按用户维度统计”，每个用户只查自己的。
- **中间件汇总** ：Sharding-JDBC 支持自动聚合 `count` 、 `sum` 、分页，但数据量大时依然慢。
- **冗余表** ：比如专门建一张“商品日销量表”，订单生成时同步写这张表（冗余）。查销量直接查它，不用跨片。代价是数据一致性要自己保证。

### 坑三：分布式事务——订单没创建，库存却扣了

这是最疼的坑。下单时要同时写订单库和库存库，不在同一个数据库，就可能出现“订单创建成功，库存没扣”（超卖）或“库存扣了，订单没创建”（用户付不了款）。

我听一个朋友说，他们生鲜电商大促时，因分布式事务没处理好，100 多笔订单库存扣了但订单没生成，用户付了钱却看不到订单，客服被骂到集体想辞职，最后赔了 5 万块优惠券。

**怎么解决？三个方案，按需选择：**

1. **本地消息表** （最简单）：在订单库建一张“消息表”，创建订单时同时插入一条消息（同一个本地事务）。定时任务扫描消息表，通知库存系统扣库存。库存系统成功后回写状态。有延迟（秒级），适合积分、日志等非核心场景。
2. **TCC 模式** （最严谨）：Try 阶段预操作（订单状态“待确认”，库存“冻结”），Confirm 阶段确认，Cancel 阶段回滚。需要写三个接口，代码量大，适合支付、下单等核心场景。
3. **Seata AT 模式** （平衡方案）：用阿里开源的 Seata，只要加一个 `@GlobalTransactional` 注解，框架自动管理事务。它通过记录 `undo_log` 表实现回滚，简单，但性能稍有损耗。

建议：积分、优惠券用消息表；下单支付用 Seata 或 TCC。

## 四、从零到一落地分库分表（附 Java 代码）

说了这么多，到底怎么动手？我分四步走，每一步都给你能跑的代码。

### 第一步：确定要不要分

别瞎折腾。如果你的数据量 < 500 万，QPS < 1000，先优化 SQL、加索引、加缓存。

**分库分表启动红线：**

- 单表 > 1000 万，且简单查询 > 500ms
- QPS > 2000，且连接数经常打满
- 预计 6 个月内数据翻倍

满足任意一条，你就该准备了。

### 第二步：选中间件——Sharding-JDBC（Java 首选）

我强烈推荐 **Sharding-JDBC** ，它是以 JAR 包形式嵌入你的应用，没有额外网络开销，性能极好。

加依赖（Spring Boot [3.x）：](http://3.x）：)

```xml
<dependency>
    <groupId>
            org.apache.shardingsphere
          </groupId>
    <artifactId>shardingsphere-jdbc-core-spring-boot-starter</artifactId>
    <version>5.5.0</version>
</dependency>
```

配置 `              application.yml            ` ：

```yml
spring:
  shardingsphere:
    datasource:
      names:db0,db1
      db0:
        type:
            com.zaxxer.hikari.HikariDataSource
          
        driver-class-name:
            com.mysql.cj.jdbc.Driver
          
        jdbc-url:jdbc:mysql://localhost:3306/order_db0?useSSL=false
        username:root
        password:123456
      db1:
        type:
            com.zaxxer.hikari.HikariDataSource
          
        driver-class-name:
            com.mysql.cj.jdbc.Driver
          
        jdbc-url:jdbc:mysql://localhost:3306/order_db1?useSSL=false
        username:root
        password:123456
    rules:
      sharding:
        tables:
          orders:
            actual-data-nodes:db$->{0..1}.orders_$->{202401..202412}
            database-strategy:
              standard:
                sharding-column:user_id
                sharding-algorithm-name:db-by-userid
            table-strategy:
              standard:
                sharding-column:create_time
                sharding-algorithm-name:table-by-month
        sharding-algorithms:
          db-by-userid:
            type:INLINE
            props:
              algorithm-expression:db$->{user_id%2}
          table-by-month:
            type:INLINE
            props:
              algorithm-expression:orders_$->{
            java.time.format.DateTimeFormatter.ofPattern("yyyyMM").format(create_time)}
```

解释： `orders` 表按 `user_id` 模 2 分到 `db0` 或 `db1` ，再按 `create_time` 的月份分到 `orders_202401` 、 `orders_202402`... 等表。

写代码时完全不用改：

```java
@Mapper
public interface OrderMapper {
    @Select("SELECT * FROM orders WHERE user_id = #{userId} AND create_time BETWEEN #{start} AND #{end}")
    List<Order> listByUser(@Param("userId") Long userId, 
                           @Param("start") LocalDateTime start,
                           @Param("end") LocalDateTime end);
}
```

Sharding-JDBC 会自动把 SQL 路由到正确的库和表。

### 第三步：平滑迁移数据（不停机）

迁移最怕丢数据、停业务。我推荐 **双写 + 灰度切换** 方案。

1. **双写阶段** ：代码里同时写旧表和新分片表，读还是读旧表。运行一两天，新数据逐渐追平。
2. **校验阶段** ：写脚本对比新旧表数据行数和关键字段汇总，确保一致。
3. **读切换** ：把读流量切到新表，旧表还保留双写。观察一天，没问题就进入下一步。
4. **停写旧表** ：删除代码中的旧表写入逻辑，旧表不再使用。再过几天，备份后删除旧表。

我写过一个简易的数据校验脚本，用 Java 对比两个数据源：

```java
@Component
publicclass DataValidator {
    @Autowired
    private JdbcTemplate oldJdbc;
    @Autowired
    private JdbcTemplate newJdbc;

    public void validateOrderCount() {
        Long oldCount = 
            oldJdbc.queryForObject(
          "SELECT COUNT(*) FROM orders", Long.class);
        Long newCount = 
            newJdbc.queryForObject(
          "SELECT COUNT(*) FROM orders", Long.class);
        if (!
            oldCount.equals(newCount))
           {
            
            log.error(
          "数据量不一致！old={}, new={}", oldCount, newCount);
            // 触发告警
        }
    }
}
```

### 第四步：扩容（从2库变4库）

当数据量再增长，需要扩容。用 **一致性哈希** 减少数据迁移量。

Sharding-JDBC 支持动态扩容，但最简单的方式是：

1. 新建两个新库 `db2` 、 `db3` ，结构相同。
2. 使用迁移工具（如 `ShardingSphere-Scaling` ）把旧分片的数据按新规则（如 `user_id % 4` ）重新分配到 4 个库。
3. 迁移完成后，修改配置 `algorithm-expression: db$->{user_id % 4}` ，重启应用。

扩容期间依然可以双写，业务不中断。

## 五、监控：别等出事了才发现

上了分库分表，必须监控这些指标：

- 各分片 QPS 和响应时间（用 Prometheus + Grafana）
- 数据倾斜度（最大分片数据量 / 最小分片数据量，超过 1.5 要告警）
- 分布式事务失败率（超过 0.01% 要排查）

Spring Boot Actuator + Micrometer 可以暴露这些指标：

```java
@RestController
public class MonitorController {
    private final MeterRegistry meterRegistry;
    private final Map<String, AtomicLong> shardQps = new ConcurrentHashMap<>();

    public void recordQuery(String shard) {
        
            shardQps.computeIfAbsent(shard,
           s -> 
            meterRegistry.gauge(
          "
            shard.qps"
          , new AtomicLong(0)))
               .incrementAndGet();
    }
}
```

## 六、结尾：你可以不用，但不能不会

分库分表不是万能药，它增加了系统的复杂度。但当你真的遇到千万级数据、亿级流量，它就是绕不开的坎。

最后送你三句话：

1. **别过早优化** ：单表能扛就先扛着，扛不住了再分。
2. **分片键是灵魂** ：选错了，后面全是坑。
3. **分布式事务别硬撑** ：能用消息表就别上 TCC，能最终一致就别强一致。

三年前我被奶茶嘲讽，三年后我喝着咖啡看监控面板上平滑的曲线，心想：当初要是早点搞，那杯奶茶我也不用浪费了。

希望你的数据库，永远用不上分库分表。但如果用得上，希望这篇文章能帮你少掉几根头发。
