今天在处理老项目的遗留问题时，发现数据库中记录的数据存在冗余数据，然后发现这些数据是存在问题的，因为这个表记录了用户在办理业务成功还是失败的表，情况无非两种要么成功，要么失败，但是数据上出现了业务办理失败了，但是成功才会存入的数据也展示处出来了。

![image-20220801184009636](image-20220801184009636.png)

就像上边倒数第二条属于一样，经过分析，发现是用户在提交成功之后页面跳转，然后返回上一步操作，这时客户拿着之前操作完的订单号再次操作了这条数据，因为这条数据是调用接口的，接口方再次处理这个数据时就会报错，操作的就只用result和msg字段。所以造成操作数据，之后优化了代码，

```java
    String orderId = get("orderId");
        Record order = service.findOrderById(orderId);
        if(order == null){
            renderJson(Ret.fail("msg","处理异常！原因：未查询到订单信息，请重新操作！"));
            return;
        }
        //查看实名制是否通过
        if("1".equals(order.getStr("SMSTATUS"))){
            renderJson(Ret.fail("msg","实名制未通过，请实名制通过后再次尝试！"));
            return;
        }
        //如果当前订单结果和描述不为空，则不允许提交
        if(StrKit.notBlank(order.getStr("CRM_RESULT")) || StrKit.notBlank(order.getStr("CRM_RESULTMSG"))){
            renderJson(Ret.fail("msg","您已提交过订单，请耐心等待！"));
            return;
        }
```

校验了下数据库中的result和msg，如果这两个元素为空则必然没被提交过，但是这样处理过后发现还是会存在数据冗余的问题，因为在调用接口完成操作后，会回到最初开始的页面，由于用户在操作这些流程的时候好多都是不仔细看的，如果提示信息简单看后又回到了开始页面，大多数人都是会在操作一次这个流程，但是这时就出现了冗余数据，并且客户体验不好的问题，由于功能是链接到app上的并灭有特别好的办法，在处理玩业务后直接关闭窗口，所以功能就暂时这样了，所以在写代码时一定要考虑周到，表单重复提交的问题，一定要校验，数据量小，可以对照日志进行修改，否则数据量上来了，造成的损失可以大了！
