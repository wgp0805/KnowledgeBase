### 起因

总结一次，经验教训，今天是周末前的最后一天，今天的心情也和这天气一样，上午就连下班前都是风和日丽的，直到下班前，一个投诉打破了这部分美好，让自己增加了一次惨痛的教训，由于自己的疏忽大意，造成了线上事故，而且这个事故也发生了好久，知道今天才被发现，下面就简单属下这个事故吧，也算是给自己一个教训，一个警钟，

### 事故分析

1. 上班时突然接到投诉，说本次的办理功能该校验的东西没有进行校验，当时我就再想这个功能，我明明测试过的，为什么这里的校验失效了，找到代码后发现，本来接口我记忆中应该返回的是`true` 和`false`,但是这里为什么是返回的List

![code](image-20230519182959514.png)

一开始认为是别人提交了接口的代码导致我此处的校验失败，但是发现提交记录是自己提交的，这时我就有点慌了，为什么会这样呢，这时发现了问题，赶紧修改了代码传到线上，怕后来的办理还是出现这个问题

2. 然后继续抱着侥幸的心理，心想没准是对端接口返回的数据有问题，然后继续找到了日志，发现接口一切正常并没有什么问题，这时我感觉自己彻底凉了。

3. 然后突然我想起来另一个办理的业务也调用了这个接口，是不是也有这个问题啊，然后马上确认，想到这时候我死的心都有了，连辞职信怎么写都想好了，但是转念一想就算跑路走人，也得把自己挖的坑填一填，能填多少是多少呗，要不可太不讲究了，当我打开这个功能的代码时我才想明白，是因为现在要看的这个功能这里的校验有调整，所以调整了接口的回参，所以造成了接口回参不匹配，有个人不喜欢if嵌套的写法，所以if里边判断的都是不满足条件然后给用户返回提示信息，所以由于回参结果不匹配了，返回的都是`false`就默认了他没有那项拦截

   ```java
    //result2接口返回结果
   if (result2.getBoolean()) {
               renderJson(Ret.fail("msg", "错误信息"))
               return;
     }
   
   //getBoolean的校验内容
   public boolean getBoolean() {
   		return Boolean.valueOf(data.toString());
   	}
   ```

### 反思

> 更改过后涉及到的所有地方都需要进行代码检查

这次事故的主要原因就是在接口调整之后只看了当前功能造成的后果，马虎是要付出沉痛的代价的，仔细点没什么坏处

> 闲着没事的时候学习新技术的同时也要做代码的 review

没事回看下自己的代码终归是有好处的，写完就不看永远也不知道自己的问题在哪，review就显得尤为重要，==谨记==，没事了不是只有学习新技术一种提升方式的。

> if嵌套的写法和顺序的写法各有各的好处但是一定要谨慎

if嵌套时的好处在于，如果当前条件不满足时，就不会进入后续的逻辑业务当中，可能代码会直接报错，但是造成损失可能较小，但是可读性差，但是这不是绝对的，根本原因还是在于马虎造成的判断问题

```java
if(条件){
	if(条件){
		核心业务
	}
}
```

顺序写法

```java
if(校验条件){
	错误信息
	return;
}
if(校验条件){
	错误信息
	return;
}
核心业务
```

这样的好处时代码可读性好，业务清晰，但是在判断的时候就需要反着判断，满足条件就是`false`,不满足就是`true`，相对结合业务的时候比较抽象，容易犯错，综合所有我还是选择了这种方式，这样做可以让代码简洁一些，但是马虎的问题就会呗放大，所以归根到底，还是自己马虎的问题造成的过错，用if嵌套的方式固然好理解，但是开发工作不是一个人的事情，写的乱七八糟是会被骂的。
