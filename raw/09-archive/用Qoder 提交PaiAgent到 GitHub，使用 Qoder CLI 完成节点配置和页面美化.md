大家好，我是二哥呀。

[上一篇](https://mp.weixin.qq.com/s/6dETHWWRzXzp1ECrxvKItw) 我们用 Qoder 把 PaiAgent 的脚手架搭好了，前后端的通信、画布中的工作流，全都顺利跑起来了。但离交付的产物还有一段距离，比如说：

- 工作流没有输入输出节点，没办法将用户的输入和最终的 AI 播客输出承接起来。
- 对应节点也没有配置项，比如说 LLM 节点要引用输入节点的文本作为输入参数，超拟人合成节点要引用 LLM 的输出作为输入参数，输出节点要引用超拟人合成节点的输出，并且输出作为回答内容呈现给用户
- 整体的界面也比较原始风

所以这篇内容主要包括：

- 使用 Qoder 将 PaiAgent 提交至 GitHub，把项目真正工程化
- 使用 Qoder CLI 新增 input 和 output 节点，让工作流闭环
- 使用 Qoder CLI 新增节点配置，让它们变得可调、可控、可扩展
- 调试输入输出节点，让整个流程能跑通；有 bug 直接让 Qoder CLI 来改
- 使用 Qoder 智能会话美化界面，让它成为一个真正可以交付的产品

## 一、将 PaiAgent 源码提交至 GitHub

我们把脚手架搭完之后，接下来要做的事一定是把代码推到 GitHub。

为什么呢？

第一，防止源码丢失，万一本地误删了，AI 辛辛苦苦肝的代码全没了，他可是会哭爹喊娘的。

第二，每个阶段的代码都是可以一键回档的，万一 AI Coding 嗨了偏离我们预期的方向，可以及时版本回溯。

第三，开源到 GitHub 的另外一个好处就是，大家在学习的时候有源码依据。

Qoder 内置了完整的源代码管理能力，不需要我们手动敲 git init、git add、git commit这些基础命令。

点一下左侧的【源代码管理】小图标，接着点【初始化仓库】，相当于执行了 git init。

![](https://cdn.paicoding.com/paicoding/0b86f93b9119ca285ea903ae341e5f3a.png)

点击【生成提交信息】的小图标，还可以让 Qoder 帮我们生成有意义的 commit 说明。

![](https://cdn.paicoding.com/paicoding/70821ef0460fe0d5b5ed03aca9d4790f.png)

接着点【提交】，这里会看到所有待提交的文件。

![](https://cdn.paicoding.com/paicoding/280bfa7b0c9a40998f81a9bff51cd1e8.png)

Qoder 编辑区可以看到更详细的提交文件，这次搭骨架，足足有 84 个文件。

![](https://cdn.paicoding.com/paicoding/6c2d245cfd46d0efb508389fe111f363.png)

在 commit\_editmsg 的右侧有一个对号，点一下这里就提交上去了。

![](https://cdn.paicoding.com/paicoding/3c7fd2349f837a3cae3fe47de339ad76.png)

接着点【发布 branch】，也就是把刚刚的暂存推送到 main 分支。

![](https://cdn.paicoding.com/paicoding/1bc64d4d382018c14ab58831eb55dbc2.png)

如果之前没有登录过 GitHub，这里需要一个授权操作，拉到底部，点击【general token】就好了。

![](https://cdn.paicoding.com/paicoding/04fd5c22f3f646cad32ea27d6f6c5b63.png)

回到 Qoder 粘贴刚刚生成的 token，按下 enter 键，就可以将代码发布到 GitHub 上了。Qoder 会给一个提示信息，点击可以跳转到 GitHub，看到刚刚提交的代码： [https://github.com/itwanger/PaiAgent](https://github.com/itwanger/PaiAgent)

![](https://cdn.paicoding.com/paicoding/bbeb698b9ae3e405c873607f41ef9b43.png)

## 二、使用 Qoder CLI 新增 input 和 output 节点

代码已经安全托管到 GitHub 上了，接下来我们可以心无旁骛地：用 Qoder CLI 给 PaiAgent 加上 input 和 output 节点。

点击 Qoder 的【切换面板】，调出终端，然后输入 qodercli，就可以进入 Qoder CLI 的主界面了。

![](https://cdn.paicoding.com/paicoding/8c69d3d9f99c3e6cc08c0123fcb7e3df.png)

以前聚的命令行是极客们才敢用的编程方式，现在倒反天罡了，感觉自己什么都能干，感觉自己就是一名极客，并且这种感觉非常好，非常好。

我们先输入 `/init` 让 Qoder 读一下项目上下文，让 Qoder 记住架构和业务逻辑。

![](https://cdn.paicoding.com/paicoding/3357c41bf9a92ddd01ced377c6b35b3f.png)

新手记不住命令不要紧，你按下任何一个字符，Qoder CLI 都会实时给你相关的提示说明。

稍等片刻，Qoder 就会生成一份项目文档，名为 AGENTS.md。

接着，我们就可以把需要做的事情告诉 Qoder CLI，比如说：

> 在画布上，没有默认的 input 和 output 节点，没办法将用户的输入和最终的 AI 播客输出承接起来。

它就会自己分析需求，列出开发计划，然后一个一个去完成。

我们需要做的，静静看它表演，哈哈。

如果有任何问题，可以及时反馈给它，比如说：

> 我不需要新增输入输出节点分组,只需要在画布的编辑区域默认添加好一个输入,一个输出节点,懂我的意思吗,因为他们是必须的,不需要手动添加.

然后很快就会把修改反馈给我们。

OK，确认没问题。

## 三、使用 Qoder CLI 新增节点配置

在继续第三小节的内容之前，我想说说 Qoder CLI 这种命令行编程的优势。

命令行编程真正的魅力不在于“极客”这种空洞的赞美，而是它的反馈是及时的，节奏是连贯的。

整个开发体验更接近 Vibe——你输入需求，它马上去理解、拆解、回应你，就好像你在和 AI 对话、沟通、交流，两个人一起编程。

好，我们继续。

选中输入节点的时候,在节点配置这里,会显示输入节点的配置信息,变量名为 user\_input 变量类型为String,描述为用户本轮的输入内容,是否必要为必要（勾选）

回到预览页面，确认一下没问题。

继续提需求：

选中输出节点的时候,节点配置这里,有一个输出配置,点击添加,添加一行表单,第一个为输入框,作为参数名,第二个为参数类型,有两个类型,输入和引用,选择输入的时候,第三个为输入框,手动输入;选择引用的话,可以引用之前节点的输出;然后有一个回答内容配置,里面可以将输出配置中的参数用{{}}引用过来

有没有发现，Qoder 的响应非常准确，基本上我们预期什么，就会交付什么给我们。

有任何需求，都可以直接提给 Qoder CLI，就按照自然语言的形式口述给他。

节点之间的连线没有箭头,比如说如果连接输入到输出两个节点,输出节点的连线要有一个箭头显示出来,表示从输入节点到输出节点

OK，我们给输入节点和输出节点连上线，然后将工作流保存为 qoder2。

## 四、使用 Qoder CLI 修 bug

基础的编码工作完成后，我们我们点【调试】按钮，输入文本来测试一下工作流是否能正确执行。

很遗憾，报错了。不过不要担心，我们把错误直接扔给 Qoder CLI，让它来自我修复一下。

很快，错误就修复了，并且会告诉我们错误原因：因为 input\_data 字段类型是 JSON，但代码直接传入了一个普通字符串，所以需要把 inpudata 字符串包装成 JSON 对象。

重启后端，然后再来测试一下。

OK，整个流程跑通了，可以看到每个节点的执行状态，以及数据的流转状态（由于还没有中间节点，比如说 LLM 和超拟人合成节点，所以耗时很短，忽略不计）。

好，我们继续下一个 bug 的修复。

页面刷新后,整个工作流就丢失了,我明明已经保存工作流为 qoder4,但是刷新页面后不见了.第一,保存工作流后,应该将工作流 id 在 URL 中显示出来,这样可以直接加载当前的工作流;第二,加载按钮还处在待开发,应该是要加载之前已经保存的工作流吧.

好，来继续测试下。点击【加载】按钮，的确把我们之前保存的工作流列表都展示出来了，点击【加载】确实也都能正确加载到当前画布中。

还是非常符合我预期的，说明 Qoder 的完成度非常高，能够准确识别我们的意图，并且对上下文的把控能力非常强，否则它不可能在短时间内就把问题修复，毕竟要编码呢，涉及到的代码文件多达 87 个。

确认没问题后，我们就把这一版的代码提交。

如果有其他 bug，就继续让 Qoder CLI 干活就对了。

执行工作流的时候,输出节点输出的内容应该是回答内容配置中的完整体,比如说我配置了 Qoder CLI：{{ouput}}, 那应该把Qoder CLI：也输出出来,加上 {{output}} 替换过的内容

OK，搞定。

还有一些细节我没有贴出来，基本上就是遇到什么问题直接抛给 Qoder CLI 就行，我的体感是都能帮我们解决，我们所需要的就是把控全局，告诉它我们看到的真相。

## 五、使用 Qoder 智能会话美化界面

虽然 Qoder CLI 也可以上传图片，按下 ctrl+f 键就可以唤起图片上传的功能，但还没办法直接复制粘贴（应该后续的版本会迭代）。

这里也是温馨提醒大家一点，在使用 Qoder CLI 的过程中，如果你碰到了 bug，可以输入 /feedback+ 描述向他们反馈。

Qoder 的智能会话模式下，上传图片相对简单，直接复制粘贴就行了，我们直接让它开始干活：

我感觉整个界面有点丑，能在此基础上做一些美化吗？

Qoder 智能会话模式有一个特点是，会去找之前我们的对话内容，这样上下文环境就是连贯的。

整体优化的细节还是蛮多的，包括顶部导航栏、左侧节点面板、中间画布区域、右侧配置面板、整体配色方案、动画效果等等。

OK，页面会自动刷新，再来看一下，是不是已经好多了，整体的布局版面舒服多了，有没有？

至少在我一个非专业前端的后端程序员眼里，已经非常漂亮了。

## 六、跟上 AI Coding 成长的脚步

总结一下前 5 个小节我们完成的任务，第一节我们先把源码扔进 GitHub，把那个最烦人的隐患解决掉：版本不会丢。

第二节开始动刀，在 Qoder CLI 里把 input 和 output 这两个基础节点补上。

第三节我们把节点的配置补全；第四节我们把遇到的 bug 全部修复掉，整个过程涉及到前后端代码全部由 Qoder 来完成，并且完成质量很高。第五节我们利用 Qoder 的智能会话优化了一下页面，整体更美观了。

相信大家也都亲身感受到了，自然语言正在成为新的编程语言，代码编写逐步交由 AI 来接管。我们开发者需要做的，就是洞察需求、描述意图、编排逻辑、验收结果。

Qoder 的核心定位是 Agentic Coding Platform for Real Software，这其中的两个关键词面向真实软件和智能体编程平台都在这次研发 PaiAgent 的过程中体现的淋漓尽致。

现在很多 AI Coding 写一些小 demo 都已经驾轻就熟，但真实的开发场景远不止于此，一个系统从 1 到 100，中间可能几年/十几年的沉淀、祖传代码、历史包袱、上下游依赖，全摆在那儿。

你得兼顾长期迭代、版本兼容、存量维护，还得遵守组织里的各种规范、代码风格、框架约束，对 AI Coding 的能力就要求比较高。

其次，人和 AI 之间的知识是不对齐的，如果你说不清楚，表达不明白，不能切中要害，不能在 AI 走偏的时候及时纠正他，coding 出来的东西可能就风马牛不相及。

我在和一个同学聊天的时候就提到，开发者的工程能力很重要，要和 AI 在某一个梯队，差距太大的话，是无法驾驭的。

这也是我从 0 到 1 做 PaiAgent 这个项目的一个初衷，让大家体验真正的 AI Coding，看看这个过程中我们到底需要做些什么，才能让 Qoder 按照我们的意图去编码，去实现我们要的效果。
